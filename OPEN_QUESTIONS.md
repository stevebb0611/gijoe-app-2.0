## 5. Completeness & accessory semantics — ✅ RESOLVED
Decisions (May 2026). Full math is now in `README.md` → *Completeness math* and `INSTANCE_MODEL.md` → *Completeness & the rebalance engine*.

- **"Complete" is per-instance.** A figure counts as complete when **at least one owned copy is whole** (has every required accessory at full required quantity). The KPI `FIGURES COMPLETE` and the green ✓ both mean *complete-now* (≥1 whole copy), **not** aggregate parts-on-hand.
- **Two related states, deliberately distinguished:**
  - **complete-now** — a copy is whole *as the parts are currently arranged*.
  - **completable** — the parts you own *could* make a copy whole if accessories were moved between copies (and/or pulled from the Parts Bin). This is the old aggregate `Σ min(owned, req)` reaching 100%.
- **Rebalance recommender (new feature).** When a figure is *completable but not complete-now*, the app recommends the moves to fix it — e.g. *"Move STALKER's Backpack from No. 2 → No. 1 to complete a copy."* Surfaced as a `⚖ REBALANCE` hint in the list/modal and (later) on Instance Detail. The optimizer fills copies one-at-a-time (greedy) to **maximize the number of whole copies**, then diffs that against the current arrangement to produce the move list.
- **Spares → Parts Bin.** Extra accessories beyond `required × copies` **overflow** and live as loose parts in the Parts Bin (that *is* the Bin's job). An instance never counts more than its required quantity; the overflow is deposited automatically (see `PARTS_BIN.md`). *(June 2026: the Parts Bin no longer surfaces a "surplus" count/label — once deposited a part is just loose stock; the figure-side overflow field is still named `surplus` in code.)*
- **Year shows two meters, not one.** Every year header carries **COVERAGE** (owned figures ÷ full-year roster — "how much of the year do I have") **and** **COMPLETE** (complete-now figures ÷ owned figures — "how many of those are whole"). The two goals are separate and both visible.
- **Partial quantities read as fractions.** Required-qty > 1 (e.g. "Skis ×2") shows as `Skis 1/2` with a partially-filled state and gives **proportional** credit toward a copy's completeness (1 of 2 skis = +1 of the 2 needed).

- **Accessory variant grouping (`group_id`) + `release_context` — ✅ LOCKED (June 2026).** Two structuring fields on the flat accessory blueprint: a shared **`group_id`** = interchangeable variants ("own any one" box — e.g. Scrap-Iron thin/thick activator `8401`, A.V.A.C. soft/hard parachute `8601`); **`release_context`** = `retail` counts, `convention`/`bonus`/`mail-in`/`exclusive` are tracked but excluded from completion (their own context box). Multi-piece sets are plain qty-aware rows, not a special "assembly." Footer is binary `Complete`/`Incomplete`. Full spec + visual rules in `PARTS_BIN.md` → *Accessory completeness model*; reference build is `GI Joe Tracker - Accessory Sub-Groups v2.html`.

Remaining follow-up: tune what counts toward *complete-now* for accessories you intentionally keep loose (e.g. you may store a figure and its parts separately on purpose) — likely a per-instance "stored loose / assembled" flag later. Not blocking.

## 9. Mobile scope — ✅ RESOLVED (June 2026)
**Decision: responsive web now, PWA later, never native.** The target is a single responsive web app (one codebase, one data layer) that reflows for phones — *not* a separate native build. Mobile is treated as a **property the app carries from day one**, not a later phase: build desktop-first and solid, but don't paint into a desktop-only corner.

- **Why this shape.** Native was struck — it's a second codebase to build + maintain for a single-user personal tool, with deep-phone-integration payoffs the app doesn't need. Responsive→PWA is the *same app gaining abilities*, not a rebuild, so it's the cheap path that still serves the flea-market scenario.
- **Three cheap habits enforced while building desktop** (these *are* the "do it now" cost — everything else defers):
  1. **No hard-coded pixel widths** — fluid layouts, `max-width` not fixed `width`, so screens reflow below the `900px` breakpoint instead of needing a parallel mobile layout.
  2. **44px minimum hit targets** everywhere (already a standing rule).
  3. **Modal → full-screen sheet** on narrow viewports (the detail modal and both Add flows). Also: KPI header stacks, gallery drops to fewer columns, year header sheds its two meters (the existing ≤900px rough-in is the seed — promote it from rough to real).
- **PWA is a layer added *on top*, in two independently-shippable stages — neither is built first:**
  - **(a) Install shell — cheap, add whenever desktop is responsive.** A web-app manifest (name, icon, theme color) + `display: standalone` → tap-to-home-screen, full-screen, no browser address bar. This is the flea-market "real app icon." Trivial, no architectural impact, **back in scope** (it had been lumped into the §5 "PWA out" call — see the reconciliation note there).
  - **(b) Offline reads — deferred, gated on real pain.** A service worker that caches the app shell + read responses so it loads on dead signal. Only add when a phone-at-shows workflow actually proves painful.
- **Offline *writes* / sync stay OUT (unchanged).** The backend is online-assumed (Turso/libSQL). Caching a live DB for offline *reads* is a contained later task; **writes-while-offline (offline queue, conflict resolution, IndexedDB mirror) remain a separate future project**, not part of any first PWA pass — consistent with `BACKEND_AND_SCALE.md §5`.
- **Net build order:** desktop solid (responsive, the 3 habits) → manifest/install shell (cheap, early-ish) → offline read-cache (only if flea-market pain is real) → offline writes/sync (separate project, maybe never). Define the phone patterns for the dense **row**, the instance **accordion**, and the **modal-as-sheet** during the desktop build so they reflow correctly from the start.

The production-variant layer (character → figure → variant → instance) and the physical "tell" are specified in **`VARIANTS.md`**, with a deepened §7 that resolves every load-bearing decision to a build-ready spec. **The UNIDENTIFIED / identify-later lifecycle was dropped entirely** — if a collector can't identify a figure/variant in hand it stays on the physical desk for review and is **not** added; the app only ever holds identified copies (no defer, no null-means-unknown, no in-app review queue). Net of the decisions:
- **(a) field-name collision — ✅ RESOLVED.** The word *variant* was doing five jobs; §7.1 retires the free `variant` string and locks `Figure.characterName` / `versionLabel` / `specialty`, production variants live only in `variant_lookup`, `variantCount` is derived. Includes a one-time rename + string-triage migration map.
- **(b) source files — ✅ RESOLVED.** 708-row file dropped; `figures_2.0` is the sole catalog source (a DB reference table). It carries a variant *count* but not the *tells*, so `variant_lookup` is a thin **owner-authored** table — confirmed: **everything is manually added**, no import.
- **(c) `Instance.variantId` FK — ✅ RESOLVED.** Nullable, but **null = single-variant figure only** (not "unidentified"); multi-variant figures require a variant at Add time. No audit/timestamp fields (no identify-later ⇒ no guess-history); correction via Instance Detail `change ›`. `ON DELETE RESTRICT` on variants.
- **(d) surface variants in Inventory — ✅ DONE** (pips / accordion / rail; re-wire to the real FK in §7.7).
- Knock-ons: rarity stays in data, lightly surfaced (§7.4); **completeness keys on `figureId` only — no blueprint override** (§7.5): whole-figure mail-in/convention/exclusive releases are their *own* catalog rows (the 700-block), lone channel-only accessories use `release_context`. Catalog IDs are stable keys with **intentional gaps** (figures 001–629 + 700–724; accessories in semantic blocks — see `PARTS_BIN.md`) and the catalog is **owner-extensible** (§7.5.1), splitting the not-a-clean-pick action into *append a real missing figure* vs. *custom homemade figure* (§7.6).

**Remaining = build work only** (§7.8, for Claude Code): schema rename + migration, add the FK, stand up `variant_lookup`, and re-wire the three screens off synthetic data.

The prototype runs a type system wired to CSS vars (`--font-display` = `Oswald`, `--font-mono` = `Space Mono`, `--font-hand` = `Patrick Hand`). **June 2026 cleanup:** the **Inventory page now loads only Oswald + Space Mono** — `Patrick Hand` and the `--font-hand` var were **removed there as dead weight** (the hand face was never applied on Inventory; it still appears on the *Scale States* page). Net: interactive chrome + stat labels are all Oswald (display), metadata/tags are Space Mono. Decisions to lock before/early in implementation:

## 15. Expansion to Vehicles (+ Playsets) — 🔭 FUTURE / DIRECTION SET
A planned major expansion — estimated **~3× the size** of the figures/accessories experience. The top nav is intended to become **[Figures] [Vehicles] [Parts Bin]**. A placeholder **[Vehicles] "in development"** chip now sits in the header as a visual reminder; it is non-functional. Direction agreed in design review (not yet built):

- **The decision to lock now (cheap today, expensive to retrofit): model `itemType` as first-class data.** Treat `figure | vehicle | playset` as a typed field on one shared "collectible" shape that reuses the **same completeness model** (catalog → ownership → instances → required parts). Vehicles and playsets are not a new system — they're a thing you own with required parts you're missing. Once typed, every UI question below becomes a cheap, reversible *view* choice.

- **(1) Vehicles vs. Playsets — one chip or two?** Ship **1a (a single [Vehicles] tab)** first, with **Playsets as a modifier chip inside it** (mirrors the existing Show Duplicates / Show Collection Gaps modifier group). Rationale: the vintage playset count is small, so a dedicated tab would be nav bloat. Because playset is modeled as its own `itemType`, **promoting it to a full [Playsets] tab later is a one-line nav change**, not a refactor — 1b stays available for free.

- **(2) Header stats — don't grow the grid; make them contextual.** Avoid the 2×3 and 3×3 explosions. **The three stat boxes reflect the active tab** (Figures → figure metrics, Vehicles → vehicle metrics, Playsets → playset metrics), keeping the **Unique / Total / Complete** language verbatim. This is the only option that scales with any number of item types and keeps the header visually stable. The tradeoff — no at-a-glance *whole-collection* rollup in the header — is intentionally handed to (3).

- **(3) Cross-type "By Year" overview — yes.** A unified view that aggregates **all item types by year** ("how complete is my *1985* — figures *and* vehicles?") is compelling because year (now Series, see #6) is the natural axis already grouping everything. Make it **its own mode** (a fourth nav item or a view toggle), not crammed into the type tabs. This view also **hosts the whole-collection rollup stats** that (2) deliberately keeps out of the header.

- **Knock-on:** **Parts Bin will need an item-type filter** once vehicle parts enter the pool — vehicles/playsets carry a different physical-part vocabulary (missiles, stickers, canopies, panels) than figure accessories.

Synthesis: **one shared typed model → [Figures] [Vehicles] [Parts Bin] with Playsets as an in-Vehicles filter → contextual 3-box header stats → a separate cross-type By-Year overview for rollups.**

## 17. Build readiness — immediate blockers (added June 2026)

Four gaps identified at the Claude Code handoff that must be resolved before the app can be used for real collection data. Ordered by dependency.

### 17a. Backend — app is disconnected from the database
The running app reads `catalog-data.js` (a static generated file) and writes to `localStorage`. The SQLite database (`gijoe_collection.db`) is edited independently via TablePlus and has **no connection to the app**. Changes in TablePlus are invisible to the browser; changes via the app are invisible to the DB. A local backend (small Express/Fastify server or Next.js API routes) is required to bridge them. Until this exists, the DB and the app are two separate systems drifting apart.
- **Blocking:** every other item below depends on this.
- **Decision needed:** confirm the local stack — Next.js App Router + Turso/libSQL (as `BACKEND_AND_SCALE.md` specifies) or a simpler local-only Express + better-sqlite3 first.

### 17b. Vite port — prototype is still browser-Babel compiled
The JSX files (`app-inventory.jsx`, `app-add-figure.jsx`, `damage-map.jsx`, etc.) are compiled at runtime by Babel loaded from CDN. There is no real build. `PORT_VERBATIM.md` specifies the mechanical steps: scaffold a Vite React project, swap `Object.assign(window, …)` exports for real `import`/`export`, extract `<style>` blocks to `.css` files, mount root in `main.jsx`. The visual output must be pixel-identical to the prototype — this is a compilation change, not a redesign.
- **Blocking:** a real build is required before the backend can be wired in, and before the app can be deployed anywhere.

### 17c. Per-instance data model — real collection data could be stranded
`figState()` in `wf-data.jsx` synthesizes per-copy accessory allocations from aggregate counts. There is no real `Instance` entity. If you begin adding figures to the collection now (via localStorage) and the schema is later changed to a proper per-instance model, that data will not migrate cleanly. The instance schema is already specified in `INSTANCE_MODEL.md`; the question is sequencing — lock and implement it **before** accumulating real collection data, not after.
- **Blocking:** do not populate the real collection until this is resolved. See also #1 and #13.

### 17d. catalog-data.js → DB sync — edits in TablePlus don't reach the app
`catalog-data.js` was generated once from the CSVs and is now a static snapshot. If figures or accessories are edited in TablePlus (corrections, additions), those changes don't appear in the app until `catalog-data.js` is manually regenerated. Once the backend (17a) is wired up, the app should query the DB directly and this file becomes obsolete. Until then, treat `catalog-data.js` as read-only and do not edit it while real collection data depends on it — a regeneration with changed IDs or code names will orphan owned figures stored in localStorage.
