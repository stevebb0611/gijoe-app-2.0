## 5. Completeness & accessory semantics — ✅ RESOLVED
Decisions (May 2026). Full math is now in `README.md` → *Completeness math* and `INSTANCE_MODEL.md` → *Completeness & the rebalance engine*.

- **"Complete" is per-instance.** A figure counts as complete when **at least one owned copy is whole** (has every required accessory at full required quantity). The KPI `FIGURES COMPLETE` and the green ✓ both mean *complete-now* (≥1 whole copy), **not** aggregate parts-on-hand.
- **Two related states, deliberately distinguished:**
  - **complete-now** — a copy is whole *as the parts are currently arranged*.
  - **completable** — the parts you own *could* make a copy whole if accessories were moved between copies (and/or pulled from the Parts Bin). This is the old aggregate `Σ min(owned, req)` reaching 100%.
- **Rebalance recommender (new feature).** When a figure is *completable but not complete-now*, the app recommends the moves to fix it — e.g. *"Move STALKER's Backpack from No. 2 → No. 1 to complete a copy."* Surfaced as a `⚖ REBALANCE` hint in the list/modal and (later) on Instance Detail. The optimizer fills copies one-at-a-time (greedy) to **maximize the number of whole copies**, then diffs that against the current arrangement to produce the move list.
- **Spares → Parts Bin.** Extra accessories beyond `required × copies` **overflow** and live as loose parts in the Parts Bin (that *is* the Bin's job). An instance never counts more than its required quantity; the overflow is deposited automatically (see `PARTS_BIN.md`). *(June 2026: the Parts Bin no longer surfaces a "surplus" count/label — once deposited a part is just loose stock; the figure-side overflow field is still named `surplus` in code.)*
- **Year shows two meters, not one.** Every year header carries **COVERAGE** (owned figures ÷ full-year roster — "how much of the year do I have") **and** **COMPLETE** (complete-now figures ÷ owned figures — "how many of those are whole"). The two goals are separate and both visible.
- **Partial quantities read as fractions.** Required-qty > 1 (e.g. "Skis ×2") shows as `Skis 1/2` with a partially-filled state and gives **proportional** credit toward a copy's completeness (1 of 2 skis = +1 of the 2 needed).

- **Accessory variant grouping (`group_id`) + `release_context` — ✅ LOCKED (June 2026).** Two structuring fields on the flat accessory blueprint: a shared **`group_id`** = interchangeable variants ("own any one" box — e.g. Scrap-Iron thin/thick activator `8401`, A.V.A.C. soft/hard parachute `8601`); **`release_context`** = `retail` counts, `convention`/`bonus`/`mail-in`/`exclusive` are tracked but excluded from completion (their own context box). Multi-piece sets are plain qty-aware rows, not a special "assembly." Footer is binary `Complete`/`Incomplete`. Full spec + visual rules in `PARTS_BIN.md` → *Accessory completeness model*; reference build is `GI Joe Tracker - Accessory Sub-Groups v2.html`.

Remaining follow-up: tune what counts toward *complete-now* for accessories you intentionally keep loose (e.g. you may store a figure and its parts separately on purpose) — likely a per-instance "stored loose / assembled" flag later. Not blocking.

## 9. Mobile scope — ✅ RESOLVED (June 2026)
**Decision: responsive web now, PWA later, never native.** The target is a single responsive web app (one codebase, one data layer) that reflows for phones — *not* a separate native build. Mobile is treated as a **property the app carries from day one**, not a later phase: build desktop-first and solid, but don't paint into a desktop-only corner.

- **Why this shape.** Native was struck — it's a second codebase to build + maintain for a single-user personal tool, with deep-phone-integration payoffs the app doesn't need. Responsive→PWA is the *same app gaining abilities*, not a rebuild, so it's the cheap path that still serves the flea-market scenario.
- **Three cheap habits enforced while building desktop** (these *are* the "do it now" cost — everything else defers):
  1. **No hard-coded pixel widths** — fluid layouts, `max-width` not fixed `width`, so screens reflow below the `900px` breakpoint instead of needing a parallel mobile layout.
  2. **44px minimum hit targets** everywhere (already a standing rule).
  3. **Modal → full-screen sheet** on narrow viewports (the detail modal and both Add flows). Also: KPI header stacks, gallery drops to fewer columns, year header sheds its two meters (the existing ≤900px rough-in is the seed — promote it from rough to real).
- **PWA is a layer added *on top*, in two independently-shippable stages — neither is built first:**
  - **(a) Install shell — cheap, add whenever desktop is responsive.** A web-app manifest (name, icon, theme color) + `display: standalone` → tap-to-home-screen, full-screen, no browser address bar. This is the flea-market "real app icon." Trivial, no architectural impact, **back in scope** (it had been lumped into the §5 "PWA out" call — see the reconciliation note there).
  - **(b) Offline reads — deferred, gated on real pain.** A service worker that caches the app shell + read responses so it loads on dead signal. Only add when a phone-at-shows workflow actually proves painful.
- **Offline *writes* / sync stay OUT (unchanged).** The backend is online-assumed (Turso/libSQL). Caching a live DB for offline *reads* is a contained later task; **writes-while-offline (offline queue, conflict resolution, IndexedDB mirror) remain a separate future project**, not part of any first PWA pass — consistent with `BACKEND_AND_SCALE.md §5`.
- **Net build order:** desktop solid (responsive, the 3 habits) → manifest/install shell (cheap, early-ish) → offline read-cache (only if flea-market pain is real) → offline writes/sync (separate project, maybe never). Define the phone patterns for the dense **row**, the instance **accordion**, and the **modal-as-sheet** during the desktop build so they reflow correctly from the start.

## 20. TablePlus vs. live server — concurrent SQLite writes can silently fail — ✅ RESOLVED (July 2026)

`gijoe_collection.db` was in SQLite's default rollback-journal mode (`PRAGMA journal_mode` = `delete`), which needs a brief exclusive lock to commit a write — editing `quantity_required` in TablePlus and hitting Cmd+S showed no blocking error, but the write never landed while the server's `better-sqlite3` connection was live.

**Fixed: switched to WAL journal mode.** Applied directly to the live db file (`PRAGMA journal_mode=WAL;`, confirmed via `sqlite3 gijoe_collection.db "PRAGMA journal_mode;"` → `wal`) without needing to restart the already-running server — WAL is a file-level setting, and the live server kept serving (`/api/catalog` → 200) straight through the switch. Also added `db.pragma('journal_mode = WAL')` to `server/db.js` so any fresh connection (new deploy, fresh clone) sets it too; harmless to re-run since WAL is sticky on the file. TablePlus and the server can now read/write the same file concurrently without lock contention.

## 21. Live DB has no backup or version control — ✅ PARTIALLY RESOLVED (July 2026, local backups)

`gijoe_collection.db` (repo root) is the actual live/authoritative state of the whole collection — every UI write (Parts Bin damage tracking, instance/accessory edits, notes) plus every one-off migration script (`set-category-bonus.mjs`, `set-accessory-context.mjs`, `add-figure.mjs`, `set-match-key.mjs`, and direct SQL edits like the 2026-07-12 Psyche-Out `.45 Assault Pistol` recolor) lands there directly.

**The problem:** `*.db` is gitignored (`.gitignore:7`) and `git log` shows zero commits ever touching the file — none of this cumulative work is version-controlled or backed up. It exists only on this one machine's disk.

The original seed CSVs (`gijoe_db_accessories.csv`, `gijoe_db_figures_2.0.csv`, `gijoe_db_figures_accessories_group_id.csv`, `gijoe_db_figures_coo.csv`) that `server/seed.mjs` reads to rebuild the DB from scratch are a day-one snapshot that's only been manually re-synced a handful of times (COO tagging, the Quick Kick/rebalance commit, the pistol-color fix) — most DB-only migrations never made it back into them, so the two drift further apart every session. There's also a stale, **unused** `seed/` copy of these CSVs (`seed.mjs`'s `ROOT` resolves to the repo root, not `seed/`) that has already diverged from the real seed files — a landmine if anyone later assumes it's current.

**Net effect: there is no single up-to-date, human-readable "master document."** The DB is the master, but it's an unbacked-up binary file; the CSVs are stale. The Tweaks & Admin **"Export catalog (.xlsx)"** button (`server/export-xlsx.js`) is the closest thing to a live, readable snapshot — it reads straight from the DB on demand — but it produces a report, not a re-importable seed source.

The catalog CSVs already tracked there are fine (reference data), but `gijoe_collection.db` also carries the owner's per-instance notes and **bin/box storage locations** for the real physical collection — committing it would publish an inventory of a valuable collection plus where it's physically kept. Un-gitignoring and pushing the `.db` (option a) was ruled out on that basis; going private first is still on the table if the owner chooses it later, but wasn't decided this pass.

**Built instead: local-only periodic backups (option c).** `server/backup-db.mjs` (`npm run backup`) copies the live `gijoe_collection.db` into `backups/gijoe_collection_<timestamp>.db`, keeping the most recent 30 and pruning older ones. `backups/` is gitignored (`.gitignore`, belt-and-suspenders alongside the existing `*.db` glob) — snapshots exist only on this machine, same exposure profile as the live DB itself, nothing new pushed anywhere. This is a manual, on-demand command (run it before risky operations, or periodically) — no OS-level scheduling was set up.

**Remaining gap:** these backups are still single-machine, same-disk — a drive failure loses both the live DB and every snapshot. Real off-machine backup needs either (i) the GitHub repo going private so option (a) reopens, or (ii) pointing `backups/` (or a copy step) at some off-machine location (external drive, private cloud folder) — not done this pass.

**Also built: a catalog-only readable snapshot (a scoped-down option b).** The owner's tagging pass turned out to be **ongoing, not a one-time gate** — reframed the "wait until finished" plan as "make resync a repeatable command instead." A full seed-CSV replacement is still parked (see below), but `server/export-catalog-snapshot.mjs` (`npm run export-catalog`) gives the "single up-to-date, human-readable master document" this section originally said was missing: it dumps the 13 catalog/reference tables (`figures`, `accessories`, `figure_accessories`, `variant_lookup`, `figure_coo`, `series`, `factions`, etc. — deliberately **excludes** `instances`/`instance_accessories`/`accessory_inventory`/`locations`/`wishlist`, the owned-collection and personal-preference tables) into a fresh, dated `exports/<timestamp>/*.csv` + `manifest.json`, via `PRAGMA table_info` introspection (no hardcoded column lists to drift). Verified: all 13 CSVs round-trip cleanly through `csv-parse` with no column-count mismatches. Idempotent and safe to run as often as wanted (e.g. after every tagging session) — it's read-only against the DB and **never touches or replaces** the actual seed CSVs `seed.mjs` reads. Since it's catalog-only (no personal notes/bin-box data), `exports/` was left **trackable** in git (not gitignored) unlike `backups/` — the owner hasn't committed it yet, that's their call.

**Not built — still needs a scoping pass:** replacing the actual root-level seed CSVs (`gijoe_db_figures_2.0.csv` etc.) so `seed.mjs` could re-import them faithfully. That's a harder, lossy round-trip — `seed.mjs` collapses `release_context` free text into a bucket + folds the detail into `notes`, and merges multiple per-variant CSV rows into one `figures` row — reversing that cleanly enough not to introduce bad data needs field-by-field design, not a mechanical dump. The stale, unused `seed/` folder (duplicate, already-diverged copy of the root CSVs) is unaffected by this pass and still a landmine — revisit together with the full seed-CSV replacement.
